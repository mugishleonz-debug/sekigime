結婚式二次会(約40名・8卓)の席決め抽選アプリ。
本ドキュメントは**追加開発者向けの完全な引き継ぎ資料**。使い方(非開発者向け)は [guide.html](./guide.html) を参照。

- 本番URL: https://mugishleonz-debug.github.io/sekigime/ (GitHub Pages, `main` ブランチ直配信)
- リポジトリ: https://github.com/mugishleonz-debug/sekigime
- サーバー: https://sekigime.mugishleonz-debug.deno.net (新Deno Deploy / app.deno.com, アプリ名 `sekigime`)

## 1. 設計思想(なぜこう作ったか)

この順に優先している。**変更を入れるときは必ずこの優先順位に照らすこと。**

### 1-1. 「DBを持たない」が最重要の出発点
発注者の要件は「大規模なDBを持ったプログラムにはしたくない」。そこで抽選結果を**どこにも保存しない**設計にした。結果は「解禁コード(シード)+名簿」から**毎回その場で計算**される。

> **中心となる不変条件(インバリアント):**
> `同じ CODE × 同じ GUESTS × 同じアルゴリズム ⇒ 全端末・全時刻で同一の席割り`

この決定論性がすべての機能の土台:
- **遅刻者・電池切れ**: いつ・誰の端末で開いても同じ結果 → 救済ロジック不要
- **同時アクセス40台**: 計算は各端末内 → サーバー負荷ほぼゼロ
- **サーバー障害**: コード口頭発表(予備手段)だけで完走できる

**だから「抽選結果をサーバーに保存する」「Math.random()を使う」変更は絶対に入れないこと。** 決定論が壊れる = 端末ごとに結果がズレる = 会場が大混乱。

### 1-2. サーバーは「合図とチクリ」だけ
後から追加した極小サーバー(`server/main.ts`)が持つ状態は2つだけ:
1. `unlocked` — 幹事が解禁ボタンを押したか(bool 1個)
2. `attempts` — フライング(不正解錠の試み)の {名前, 時刻} ログ

抽選そのものには一切関与しない。サーバーが死んでもアプリは完全動作する(単独モード)。この「サーバーはあくまでオプション」という関係は維持すること。

### 1-3. 演出はUXの本体(飾りではない)
これは業務システムではなく**余興**。ドクロの威嚇、怒りの煽り文句、スロット、紙吹雪、検挙スタンプは仕様の中核。機能追加時も「会場で笑いが起きるか」を判断基準にする。文言のトーンは「芝居がかった脅し文句(〜だぞ、覚悟しろ)」で統一。

### 1-4. 幹事の運用コストを最小に
幹事が編集するのは `index.html` 冒頭の `CONFIG` オブジェクト**1か所だけ**。ビルド工程なし、npm なし、1枚のHTML。この「編集箇所は1か所」原則を守ること(設定を増やすなら CONFIG に足す)。

## 3. アプリ本体(index.html)の構造

### 3-1. CONFIG(幹事が触る唯一の場所)

```js
const CONFIG = {
  SERVER: "",        // サーバーURL。"" なら単独モード(ポーリングも通報も無効)
  CODE: "0705",      // 解禁コード = 抽選シード。★変えると結果が変わる
  TABLES: 8,
  GUESTS: [ {n:"佐藤 健太", g:"M"}, ... ],  // g: "M"/"F"。画面には出さない(内部制約専用)
};
```

- 人数は `TABLES × 5` ちょうどでなくてよい。`drawTables()` が各卓にできるだけ均等に割る(39名 → 5×7卓+4×1卓)。
- 現在の名簿は**サンプル(男20/女20)**。実名簿は未投入。
- **性別をUIに表示しない**のは意図的(晒す必要がない)。

### 3-2. 抽選アルゴリズム(`drawTables`)

```
シード文字列 = CODE + "#" + salt   (salt = 0,1,2,...)
xmur3(シード文字列) → mulberry32 → Fisher–Yates シャッフル → 卓サイズ配列で分割
→ 全卓に男性が1人以上いれば採用、いなければ salt+1 で引き直し(上限5000)
```

- 「女性だけの卓を作らない」は要件。判定は `tab.some(p => p.g === "M")`。
- 引き直しも salt 順で決定論的 → 全端末一致が保たれる。**制約を追加するときも必ずこの「決定論的リトライ」パターンで書く**(後述 §8)。
- 男性が7人以下など制約充足が不可能な場合は `null` を返し、画面にエラーを出す。
- PRNGを標準の `Math.random` にしないこと(シード指定不可)。xmur3/mulberry32 は実績ある軽量実装で、変更禁止(1文字変えると全結果が変わる)。

### 3-3. 画面遷移(すべて `section.screen` の表示切替、SPA的)

```
#who(名を名乗れ)→ #lock(ドクロ封印)→ #roll(スロット)→ #reveal(結果)
                        │                                    ├→ #pick(名前選び直し/他人の検索)
                        ├→ #admin(幹事コンソール、PIN制)      └→ #all(全卓一覧)
```

- `show(id)` が唯一の遷移手段。`.screen.on` だけが表示される。
- 起動時の振り分けは `boot()`: 名前未登録→#who、解錠済み(sessionStorage)→即スロット、それ以外→#lock。

### 3-4. クライアント側の永続化(すべてWeb Storage、これもDBレス思想の一部)

| キー | 置き場所 | 用途 |
|---|---|---|
| `sekigime_me` | localStorage | 名乗った自分の名前 |
| `sekigime_firstopen` | localStorage | 初回アクセス時刻(検挙の証拠) |
| `sekigime_attempts` | localStorage | 不正解錠の試行時刻の配列(検挙の証拠) |
| `sekigime_code` | sessionStorage | 解錠済みフラグ(リロード対策。タブを閉じると封印に戻る=リハーサルに便利) |

### 3-5. サーバー連携(SERVER 設定時のみ)

- **ポーリング**: #lock 表示中だけ 5秒間隔で `GET /state`。`unlocked:true` で `doUnlock()`。
  40台 × 0.2req/s = 8req/s 程度。Deno Deploy 無料枠で余裕。
- **通報**: 不正コード入力時に `POST /attempt {name}` を fire-and-forget(失敗は握りつぶす。ローカル記録が主、通報は従)。
- 通信失敗はすべて黙殺 → 単独モードと同じ動きに落ちる(グレースフルデグラデーション)。

### 3-6. フライング検知の二重構造

1. **ローカル**: 試行を localStorage に記録 → 結果画面に「検挙スタンプ」or「良い子の証」を強制表示。サーバー不要で機能する。
2. **サーバー**: 名前つきで /attempt に通報 → 幹事コンソールの検挙リストに載る。

既知の穴(許容済み・発注者に伝達済み):
- プライベートブラウズ/データ消去でローカル証拠は消える(その場合「初回アクセス時刻」が不自然に遅くなるので雰囲気でバレる)
- 他人の名前で名乗って通報を濡れ衣にできる(#who 画面に警告文で抑止)
- パーティーの余興なのでこの程度の強度で十分、という判断。**厳密化したければ本気の認証が要るが、それはこのアプリの思想(手軽さ)に反する。**

## 4. サーバー(server/main.ts)

Deno + Deno KV。エンドポイント一覧:

| メソッド/パス | 認証 | 動作 |
|---|---|---|
| `GET /state` | なし | `{unlocked: bool}` |
| `POST /attempt` | なし | `{name}` を時刻つきで記録 |
| `POST /unlock` | PIN | 解禁フラグを立てる |
| `GET /attempts?pin=` | PIN | `{unlocked, attempts:[{name,ts}]}` |
| `POST /reset` | PIN | 封印し直し+記録全消去(リハーサル用) |

- PIN は環境変数 `ADMIN_PIN`(未設定時 `8888`)。**フロントのソースにPINを置かない**こと(GitHub Pagesは全世界に公開されている)。
- CORS は `*`(公開情報しか返さないため)。/attempt はスパム可能だが脅威モデル(結婚式)上許容。
- KVのキー設計: `["state","unlocked"]` と `["attempt", ts, uuid]`。attempt を1件1キーにしているのは read-modify-write 競合を避けるため(同時通報でも消えない)。

### ローカル起動と手元テスト

```sh
deno serve --unstable-kv --port 8787 -A server/main.ts
curl localhost:8787/state
curl -X POST localhost:8787/attempt -H 'content-type: application/json' -d '{"name":"テスト"}'
curl "localhost:8787/attempts?pin=8888"
```

### デプロイ(新Deno Deploy = app.deno.com)

**旧CLI(deployctl)は使えない**(新世代アカウントでは認証ページがダッシュボードへリダイレクトされるだけ)。Deno 2.x 内蔵の新CLIを使う:

```sh
cd server
deno deploy --prod            # 初回のみ対話式(ブラウザ認証、要TTY)。以降は --non-interactive 可
deno deploy env add ADMIN_PIN <秘密の4桁>   # PINの設定/変更(適用時に再起動が走る)
```

- 設定は `server/deno.jsonc`(CLIが自動生成、org/app名を保持)
- **PINをリポジトリに書かないこと**(このリポジトリは公開。過去に書いて消した経緯あり)
- **ストレージは実測で memory モード**(新Deno Deployは `Deno.openKv()` 非対応 → §4のフォールバックが作動)。含意:
  - 再デプロイ・env変更・再起動のたびに解禁フラグと検挙リストが消える → **本番当日は絶対に再デプロイしない**
  - 万一本番中に消えたら: 幹事が解禁ボタンをもう一度押す。検挙の証拠は各ゲスト端末のローカル記録(結果画面のスタンプ)が正本
  - 複数インスタンスに分かれると解禁が一部に伝わらない理論リスクあり(この規模ではまず起きない)。最終安全網は解禁コードの口頭発表
- 動作中の保存方式は `GET /attempts?pin=…` の `storage` フィールド(`kv`/`memory`)で確認できる

## 5. 運用フロー(当日)と鉄の掟

1. 事前: 実名簿を CONFIG に投入、CODE と ADMIN_PIN を決める、QR配布
2. 欠席反映は**解禁前まで**に名簿から行を削除して push
3. 当日: ゲストはQR→名乗る→ドクロ画面で待機
4. 幹事: 「幹事の方はこちら」→PIN→「全員の封印を解く」
5. 検挙リストで罰ゲーム発表
6. 電波の悪い人には CODE を口頭で伝える(予備手段)

> **鉄の掟: 解禁後に CONFIG(名簿・CODE)を変更しない。** 全員の結果が変わる。
> リハーサル後は幹事コンソールの「リセット」+各自タブ閉じで復帰。

## 7. テストとスクショの再現方法

### 抽選ロジックの回帰テスト(Node)
`<script>` 部の drawTables までを切り出して実行する方式。過去に確認した観点:
- 40/39/36名で卓サイズが均等・重複なし・女性のみ卓ゼロ
- 同一コードで結果一致(決定論)
- 男性7名(8卓に行き渡らない)で null

スニペットは scratchpad 揮発のため、恒久化するなら `test/logic_test.js` として `node:test` で書き直すのが良い(未着手、優先度高)。

### guide.html 用スクショ撮影
puppeteer-core + ローカルChromeで実画面を自動撮影した(手撮り禁止、再現性のため)。
手順: ①ローカルサーバー起動 ②`/attempt` を2件シード ③SERVER をlocalhostに差し替えたコピーを作る ④viewport 390×780 @2x で各画面を遷移しながら撮影。撮影スクリプトの流れ: who→名前クリック→lock→誤コードでangry→正コードでroll→reveal(スタンプ)→localStorage消してreveal(良い子)→all→別ページで幹事コンソール。

## 8. 今後の開発アイデアと実装指針

### やるなら筋が良いもの
- **同席/分離制約**: 「この2人は同じ卓に」「元カップルは別卓に」。実装は §3-2 の決定論的リトライに判定を1つ足すだけ。`CONFIG.PAIRS = [["A","B"]]` / `CONFIG.APART = [...]` のような宣言形式で。制約を増やすと充足失敗しやすくなるので salt 上限と null 時のエラー文言に注意。
- **卓名のカスタム**(番号でなく「モルディブ卓」等): `CONFIG.TABLE_NAMES` を足して表示側だけ差し替え。
- **新郎新婦専用演出**: 名乗りが特定の2名なら特別画面(感謝メッセージ等)。
- **効果音**: スロット・解禁時に WebAudio。iOSは初回タップ後でないと鳴らない点に注意(名乗りタップで AudioContext を作っておく)。
- **PWA化(オフライン強化)**: manifest + service worker。単独モードとの相性は完璧。
- **WebSocket/SSE化**: 解禁の伝播を5秒→即時に。Deno Deployは `Deno.upgradeWebSocket` 対応。ただしポーリングでも実用上は十分(演出的には「ばらばらっと開く」のも味がある)。

### やってはいけない/慎重にすべきもの
- 抽選結果のサーバー保存(思想 §1-1 に反する)
- `Math.random()` の使用(結果表示系の演出のみ可。スロットの回転数字には既に使っているが、**最終結果には絶対に使わない**)
- CONFIG以外に設定を分散させること
- フレームワーク導入(React等)。1枚HTMLで完結しているのが価値。ビルドが要る時点でこのプロジェクトでは負け
- 解禁コードの厳密な秘匿を目指すこと(4桁は総当たり可能だが、間違いコードは「別の偽結果」ではなく単に開かないだけ+通報されるので、実害はフライング検挙が増えるだけ=むしろ罰ゲームが盛り上がる)

### 既知の小さな負債
- `test/` が無い(§7参照。最優先)
- angryLines(煽り文句)がハードコード。CONFIG に出すと幹事がカスタムできて楽しい
- 幹事コンソールはPIN入力のたびログインが必要(sessionStorageに保持してもよい)
- guide.html のスクショはUI変更で古くなる。§7の手順で撮り直すこと

## 9. 経緯メモ(意思決定の記録)

1. 初版: 完全サーバーレス。解禁コード=シードの決定論抽選、ドクロ封印、コード手入力解錠
2. 「40名ちょうどでない可能性」→ 卓への均等割りに一般化(SEATS設定を廃止)
3. 「フライングした人を罰ゲームにしたい」→ ローカル記録+結果画面スタンプ(サーバーレスの範囲で実装)
4. 「小さいサーバーは建ててもいい」→ Deno Deploy極小サーバー追加。幹事ボタンで全員自動解錠+名前つき通報。**このときも抽選自体はサーバーに載せなかった**(§1-1)
5. 電池切れ問題は「決定論+全卓一覧を全端末に配る」で解決済み(専用機能なし)

この順で要件が進化してきた。今後も「サーバーに載せるのは合図と証拠だけ、結果は計算で」を守れば、だいたいの追加要件はきれいに入る。
